# KB Document Design

この文書は、KB文書を初期作成・更新・整理するときの恒常的な設計ルールである。

KBはMarkdownファイルの集まりではなく、未来のAIとユーザーが再利用する知識体系として保つ。

---

## 1. 良いKB文書の条件

良いKB文書は、初見のAIが読んだときに「何を信じればよいか」が分かる。

- 冒頭で、何の正本かが分かる
- 現在有効な情報が先にある
- 目的、前提、方針、ルール、判断理由が分離されている
- 古い情報と現行方針が混ざっていない
- 未確定事項、確認事項、次アクションを本文内に抱えていない
- 他のKB文書やIssueとの役割分担が明確
- 追記の積み重ねではなく、現在の理解に合わせて再構成されている

悪いKB文書:

- 作業ログが時系列に積み上がっている
- 旧方針と現行方針が同じ階層に並んでいる
- TODOや未決リストが本文に残っている
- 同じ情報が複数文書に重複している
- 末尾追記だけで全体構造が崩れている

## 2. カテゴリ別の整理方針

| ディレクトリ | 役割 | 書く内容 |
|---|---|---|
| `company/` | 組織・事業・人・判断基準の正本 | 会社やチームの背景、ユーザーの役割、事業上の前提、意思決定の傾向 |
| `initiatives/` | 進行中の取り組み・プロジェクト文脈の正本 | 目的、現在有効な方針、設計判断、再開に必要な文脈 |
| `tools/` | ツール・SaaS・API・ローカル環境の正本 | 現在の設定、操作方法、API仕様、認証、ハマりどころ |
| `patterns/` | 複数領域で再利用できる型の正本 | 判断プロセス、進め方、設計パターン、チェック観点 |
| `reference/` | 正本ではないが参照価値のある外部情報 | 人物、資料、外部コンテキスト、具体事例 |
| `archive/` | 現行正本から外れた文書の退避先 | 旧方針、履歴、完了済みinitiative、詳細ログ |

置き場所の判断:

- ツール固有の仕様や操作方法は `tools/`
- 会社・事業・組織に依存する背景は `company/`
- 特定プロジェクトに閉じる方針や設計判断は `initiatives/`
- 複数プロジェクトに再利用できる考え方は `patterns/`
- 正本ではない外部情報は `reference/`
- 現行文書から外したが保存する価値がある旧情報は `archive/`

## 3. 初期作成の作法

新しいKB文書を作る前に、次を決める。

- この文書は何の正本か
- 想定読者は誰か
- どのカテゴリに置くべきか
- 関連する既存KBは何か
- Issueで管理すべき未決事項が残っていないか

初期作成時に、作業ログや検討メモをそのまま貼らない。再利用される知識として構造化して書く。

推奨構成:

1. この文書の役割
2. 現在有効な結論・方針
3. 前提・背景
4. 具体ルール・仕様
5. 関連KB・関連Issue

全ての文書で同じ見出しにする必要はない。重要なのは、現在の正本がすぐ分かる構造にすること。

## 4. 更新の作法

KB更新は末尾追記を原則にしない。文書全体を読み直し、現在の正本として自然な位置へ統合する。

更新時のルール:

- 古い情報は現行文書から削除する
- 残す価値がある旧情報は、別文書へ退避して `archive/` または `*_history.md` として明示的にアーカイブする
- 判断経緯は必要な場合だけ、現行方針の理由として圧縮して残す
- 未確定事項、確認事項、次アクションはIssue化する
- 現行正本に旧方針と新方針を並べて残さない
- 文書が肥大化したら、正本・履歴・設計・運用に分割する

Issue化すべきもの:

- 未決の方針
- 人間判断が必要な確認事項
- 次にやる作業
- 調査すべき疑問
- 完了条件を持つ整理・移行タスク

Issue化したら、対応する記述はdocから削除する。「→ #XX で対応」のような参照リンクも残さない。Issueはフローの正本なので、KB側に転記・参照すると二重管理になる。

KB本文に残してよいもの:

- 決まった方針
- 現在有効なルール
- 判断理由として必要な背景
- 再開に必要な圧縮された文脈

## 5. 現在地・history_logの粒度

進行中initiative docの「現在地」セクションと、`*_history.md` / `history_log` は、役割上フローを含むのが正常な領域である。一律削除はできないが、放置すると作業ナレーションが積層して肥大化する。

進行状況の正本はIssue側に置く（`issue_based_agent_workflow.md`）。docのこれらのセクションには、再開と判断再構成に必要な最小限だけを残す。

### 判定3問（残す/落とすの共通テスト）

スリム化・更新のたびに、各記述へ次を問う。3問すべてに「残す側」と答えられたものだけ残す。

1. それは**最新の状態・方針**か、過去のある時点の**出来事**か。出来事ならフロー。
2. **上書きで消えるべきもの**を、追記で積んでいないか。積んでいればフロー。
3. 同じ情報の正本が **Issue / 別doc / 引継ぎプロンプト** にあるか。あればdocから消す。

### A. 「現在地」セクション（稼働中のinitiative doc）

残す:

- 全体ステータスの最新スナップショット（領域 × 現在地 × 次の論点など）。**上書き更新**し、過去状態を時系列で積まない。
- 「直近の中断地点」を1ブロックに圧縮。対象・どこで止まったか・次の一手・再開時に効く確定事項/前提だけ。中断地点は**最新の1つだけ**。

落とす（Issueまたは別正本へ）:

- Phase別・日付別の進捗ログの時系列積層
- dry-run / 本実行の件数、投入結果の集計
- 作業ログ・引継ぎプロンプトのパス羅列、完了ナレーション
- ユーザー待ち・未決ネクスト（Issue化。過渡期は最小サマリの暫定残しのみ許容）

目安: 「現在地」全体でおおむね1スクロール（40〜60行）以内。

### B. history_log / `*_history.md`

残す:

- 確定方針・設計判断とその根拠を、年月で束ねたサマリ。日付は「いつ確定したか」の出典として付与する。
- 後から「なぜこう決めたか」を再構成するのに必要な、圧縮された経緯。

落とす（実行ログdocまたはIssueへ）:

- 実施ログ（投入件数・dry-run・スクリプト実行記録・コマンド時系列）
- 現行正本（implementation.md等）の確定方針と重複する記述。正本へ一元化する。
- 完了チェックリスト・作業ナレーション

目安: 1エントリ＝「方針＋根拠の要約」。手順・件数の再現情報は持たない。

## 6. 文書間連携

`INDEX.md` はKB全体の入口である。エントリを追加・削除したら必ず更新する。

文書間の役割分担:

- 詳細仕様は `tools/`
- 判断背景は `company/`
- プロジェクト適用は `initiatives/`
- 再利用パターンは `patterns/`
- 外部参照情報は `reference/`
- 現行正本から外れた履歴は `archive/`

各文書には、必要に応じて関連KBや関連Issueへのリンクを置く。ただし、進行状況やIssue一覧をKB側に転記しない。進行状況の正本はIssue側に置く。
