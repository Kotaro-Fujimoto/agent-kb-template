# API活用の基本動作

この文書は、エージェントが API・外部サービス（GitHub / その他 SaaS、CLI・GraphQL・REST・MCP を問わない）を操作するときに、どのAPIでも守る横断的な基本動作の正本である。

特定APIの仕様・認証・エンドポイントは各 `tools/` 文書を正とし、本文書は「どのAPIでも共通する振る舞い」だけを定める。

---

## 1. 中核となる基本動作 — 既知値で楽観実行し、失敗時だけ確認する

確認ファーストをやめる。次の順序を基本動作とする。

1. 一度確認して判明した**確定値**（ID・エンドポイント・スキーマ・スコープ・設定値）は KB に記録する。
2. 記録された既知値があるなら、**状況確認をスキップして記録値でまず操作を試みる**。
3. **失敗したときだけ**、「別口で設定が変更された可能性」を疑って再取得・確認する。

この順序にする理由（背景）:

- 「まず状況確認します → 全設定値を read → 値が取れました → 操作実行します」を毎回やると、API呼び出し回数が無駄に増え、**レート制限を超える事故**につながる。
- 既知の項目に対する確認用 read を毎回挟むと、確認で迷う間に**トークンも浪費**する。
- 既知値を記録して再利用すれば、呼び出し回数とトークンの両方を同時に減らせる。

確認は「事前の儀式」ではなく「失敗時のフォールバック」に位置づける。記録値で問題なく通る限り、確認用の read は挟まない。

---

## 2. 操作の性質で確認の要否を分ける

「確認を一律でやめる」のではない。操作の性質で扱いを変える。ここが事故防止と無駄削減の両立点。

| 操作の性質 | 基本動作 |
|---|---|
| 冪等・可逆な read／既知値の参照（ID・option・スキーマの取得など） | 記録値で即実行。確認用の事前 read は**しない**。失敗したら再取得。 |
| 不可逆・本番影響・大量の write（更新・削除・送信・一括変更） | 楽観実行の対象外。**事前確認・dry-run・バッチ分割を維持**する（§4）。 |

- 値確認（read的操作）は記録で省く。
- 不可逆 write の確認は省かない。
- 判断に迷ったら「これは可逆か」「失敗してもやり直せるか」を基準にする。やり直せないものは確認側に倒す。

---

## 3. 確定値の記録と再利用

- 一度取得して安定している値（Project ID / field ID / option ID、APIエンドポイント、オブジェクトのスキーマ、認証スコープ等）は、該当する `tools/` 文書に記録する。
- 記録値には「これは安定キーであり、まず記録値を使う。失敗したら再取得する」という前提を明示する。「念のため毎回再取得が安全」とは書かない（それが確認ファーストの温床になる）。
- 値が変わりうる条件（例: GitHub Projects の option を再編集すると option ID は新規発行される）は、記録値の近くに失敗時の再取得トリガーとして書いておく。
- 記録値が古くなって失敗したら、再取得した新しい値で `tools/` の記録を更新する。

> 具体例: GitHub Projects の安定ID（Project ID / Status field ID / Status option ID）の記録と再利用は `tools/github_projects.md` を正本とする。`/start-issue`・`/close-issue` の Status 更新は、毎回の option 取得 query を挟まず、まず記録値（プレースホルダーに置換済みの option ID）で楽観実行する。

---

## 4. レート制限・大量操作

- 大量件数を扱うときは、**1件ずつ全件 read してから処理**のような呼び出し増殖を避ける。必要な値は記録・キャッシュして使い回す。
- 一括処理は**バッチ分割**し、**1回あたりの件数上限**を設ける。全件を一度に投げない。
- レート制限（429 等）に当たったら、握りつぶさずに待機・リトライ（指数バックオフ）し、それでも続くなら中断して報告する。
- 大量の write（一括更新・一括削除・一括送信）は、**実行前に対象件数と内容を提示して確認**を取り、可能なら **dry-run**（件数・対象の事前確認）を先に行う。

---

## 5. 不可逆操作の事前確認

更新・削除・送信など、やり直しが効かない／外部に影響が出る操作は、実行前に次を確認する。

- 何を、どの対象に対して、どう変えるのか（件数・対象レコード・変更内容）。
- 可逆か。元に戻せないなら、実行前にユーザーへ確認する。
- 送信系（メール・通知・外部API送信）は、宛先・本文・件数を提示してから実行する。

可逆な read や、既知値を使った冪等な操作は、この事前確認の対象外（§1・§2のとおり楽観実行する）。

---

## 6. 認証・対象スコープの確認

- 操作前に、**どの環境（本番／テスト／サンドボックス）**に対して実行するのかを明確にする。本番に対する破壊的操作は特に慎重に扱う。
- 必要な**認証スコープ**（例: GitHub の `project` scope、各APIキーの権限）が足りているかを把握する。スコープ不足で失敗したら、不足分を提示する。
- 対象レコード／対象アカウントが意図したものか（取り違えがないか）を、不可逆操作の前に確認する。

---

## 7. エラー・失敗レスポンスを握りつぶさない

- API のエラー・失敗レスポンス・想定外の戻り値を**無視して次に進まない**。必ず内容を確認する。
- 失敗したら、原因（レート制限／スコープ不足／値の不一致／対象不在 等）を切り分けて報告する。
- 「成功したつもり」で進めない。書き込み系は、結果を検証してから完了とする。
- 整形済み出力（例: `--jq`）が壊れることがある。重要な検証は生レスポンスでも裏取りする（GitHub GraphQL の事例は `tools/github_projects.md`）。
