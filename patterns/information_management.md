# 情報管理パターン：フロー / ストック

## 基本原則
仕事の情報は2種類に分けて考える。

| 種類 | 定義 | 置き場所 |
|---|---|---|
| **フロー** | 今進行中のタスク・検討・ドキュメント。終われば消えてよい | 会話コンテキスト、一時ファイル |
| **ストック** | 次回以降も参照すべき知識・状態・経緯 | `~/agent-kb/` |

## 蒸留のタイミング
以下が発生したらフロー→ストックへ蒸留する：

- プロジェクト・タスクが完了したとき → 決定事項・現在状態を`initiatives/`へ
- ツール・システムを変更したとき → 最新状態を`tools/`へ
- 重要な方針・判断が下されたとき → 背景と結論を`company/`へ
- 同じ背景説明が2回以上出てきたとき → 上段コンテキストを`company/`へ
- 再現性のある判断・検討プロセスが見えたとき → `patterns/`へ

## 分類先の判断基準

「何の話か（トピック）」ではなく「どんな性質の知識か（型）」で分類先を決める。

| 知識の型 | 分類先 | 判断の問い |
|---|---|---|
| 特定エンティティの情報（人・サービス） | `reference/` | 後で「あの人/サービスってどんなだっけ」と調べるか？ |
| 組織内部の事業・戦略コンテキスト | `company/` | 組織の方針・事業背景の理解に必要か？（外部情報はここに入れない） |
| 再現性のある判断・アプローチ | `patterns/` | 次に似た状況が来たとき「どうするか」の参照になるか？ |
| ツール・システムの現在状態 | `tools/` | 操作・設定時に「今どうなっているか」を確認するか？ |
| 特定プロジェクトの経緯・状態 | `initiatives/` | そのプロジェクトの文脈を思い出す必要があるか？ |

## 同一経験からの多層抽出

一つの経験・活動から、複数の抽象度レイヤーを同時に切り出して格納する。

例：社外発信活動
- 具体的な人物情報 → `reference/person_name.md`
- アウトリーチで得たパターン → `patterns/outreach.md`
- 活動の事業的位置づけ → `company/context.md`

「この経験から何が学べたか」を複数の切り口で問い、それぞれ適切な粒度で別の場所に置く。

## プロジェクト完了時の処理
- **詳細は残さない**（フローなので捨てる）
- **残すもの**: 再現性のある判断の傾向・参照パターン・進め方の型
